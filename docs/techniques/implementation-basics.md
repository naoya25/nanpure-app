# 実装の前提・用語・進め方

ヒントエンジンや自動メモをコードに落とすときの **共通前提**。個別テクニックのロジックは各詳細ページを参照。

---

## 盤面とインデックス

- 81 マス・インデックス `0..80`（`lib/validates/grid.ts` の `SUDOKU_CELLS`）
- 行: `floor(i / 9)`、列: `i % 9`
- ブロック（3×3）番号 `0..8`: `floor(row/3)*3 + floor(col/3)`

---

## 確定値と候補

| 概念       | 例                                   | 備考                                                  |
| ---------- | ------------------------------------ | ----------------------------------------------------- |
| **確定値** | `0` = 空、`1..9` = 埋まり            | 現状の `grid` と同じ                                  |
| **候補**   | 各空マスに「まだあり得る数字」の集合 | ビットマスク 9bit（メモ UI と同型）または `Set<1..9>` |

機械処理の基本ループ:

1. ヒント・ユーザー入力から **確定盤** を作る。
2. 各空マスについて、**同じ行・列・ブロックに既に出ている数字を除いた集合** を候補の初期値とする。
3. テクニックは「候補の削除」または「マスへの確定（＝そのマスの候補をその数字のみに）」を返す。

---

## ユニット（unit）

行・列・ブロックのいずれか。**各ユニットはちょうど 9 マス**で、数字 `1..9` がそれぞれ 1 回ずつ現れる制約の単位。

実装では「マスインデックスの配列（長さ 9）」のリストを **27 本**（行 9 + 列 9 + ブロック 9）持てばよい。

---

## テクニック関数のイメージ（純粋）

- **候補の更新**: `(candidates: CandidateGrid, grid: Grid) → { candidates, changed }`
- **一手の提示**（UI 用）: 種類・対象マス・根拠マス・メッセージキーなどの構造体の配列

`CandidateGrid` は 81 要素。各要素は「空マスなら候補、確定マスなら無視 or 空集合」。

---

## 実装の進め方（提案）

1. **候補の初期化**を 1 関数にまとめる。
2. **伝播**（確定したマスの数字を関連ユニットから削除）を共通処理にする。
3. テクニックは **体感順または機械的な単純さ**に合わせて追加し、**単体テスト**（小さい盤面・既知の手筋）を付ける。
4. UI では「ユーザー盤」と「非表示の solution」は別用途。**ヒントは候補と論理だけ**にし、正解文字列は検証用に閉じるとチートになりにくい。
